# Design: add-podcast-transcripts

## Endpoint

`GET https://spclient.wg.spotify.com/transcript-read-along/v2/episode/{episode_id}?format=json`
with `Authorization: Bearer <web-player-token>` and `app-platform: WebPlayer` —
the direct sibling of color-lyrics on the same host, reached with the same
cookie-derived token. Host/path/query live ONLY in `api/transcripts.py`.

## The one open question: envelope shape

The exact 200-body shape is **unconfirmed** and must be captured live with the
maintainer `sp_dc` (see tasks 0.1–0.5). Two possibilities:

1. **Plain JSON** with a readable lines container → `decode_envelope` is a near
   pass-through.
2. **JSON envelope wrapping a base64 protobuf blob** (the more likely per the
   community tooling) → `decode_envelope` base64-decodes and runs a small
   hand-rolled protobuf-wire reader (varint + length-delimited fields only,
   ~30 lines, **no new dependency**). Community schema: `segments`=field 4 →
   `time_ms`=field 2 (varint), `transcription`=field 3 → `segment`=field 1
   (repeated string text).

To keep the uncertainty isolated, `decode_envelope` always returns the **same
normalized mapping** regardless of input shape:

```json
{"lines": [{"start_ms": 0, "text": "..."}, ...],
 "language": "en", "provider": null, "isAutoGenerated": true}
```

`parse_transcript` is written against this normalized mapping, so the parser and
models never change when the wire format wobbles — only `api/transcripts.py`
does. Until live capture confirms the protobuf branch, the protobuf decoder is
implemented to the community schema and covered by a known-vector unit test; the
plain-JSON branch is a pass-through. The live test is the real confirmation.

## Models

`TranscriptLine(start_ms: int, text: str)` and
`Transcript(lines: tuple[TranscriptLine, ...], language: str | None = None,
provider: str | None = None, is_auto_generated: bool | None = None)` — frozen,
slotted, on `ModelBase`, so `to_dict()`/`from_dict()` come for free by
reflection. `TranscriptLine` mirrors `LyricsLine` (same `start_ms`/`text` shape)
for one consistent mental model. No `speaker` field — the community schema has
none; add `speaker: str | None = None` only if live capture proves otherwise.
Episode-level `language`/`provider`/`is_auto_generated` are `| None` because
their presence in the envelope is unverified.

## Shared cookie provider

The lyrics token slot/builder is renamed `_lyrics_tokens`→`_cookie_tokens` and
`_lyrics_provider`→`_cookie_provider` on both clients (internal only — no public
API change). One `sp_dc` exchange then serves both `get_lyrics` and
`get_transcript`; entity extraction keeps using the anonymous token. The "no
cookies" message becomes feature-neutral.

## Open questions (carry to live verification)

- Envelope shape (plain JSON vs base64+protobuf) and the transcript key.
- Where `language`/`isAutoGenerated`/`provider` live (presumably outer JSON).
- Whether `market=from_token` / `accept-language` are required/accepted, and
  whether a public `language=` kwarg is warranted (deferred until confirmed).
- Exact non-200 split: expect 404 = no transcript, 401/403 = auth/market; add a
  403 path if observed.
- Whether transcripts are Premium/market-gated vs any logged-in `sp_dc`.
