# Proposal: add-podcast-transcripts

> **Status: targets v3.3.** Builds directly on the shipped `cookie-auth`
> capability (v3.1/v3.2). The `sp_dc` → web-player Bearer handshake, the
> `CookieTokenProvider`/`AsyncCookieTokenProvider`, and `load_sp_dc` are reused
> **unchanged** — transcripts are the second authenticated web-player feature
> reached with that same token, the sibling of color-lyrics on the same
> `spclient.wg.spotify.com` host.

## Why

Spotify's web player ships read-along transcripts for many podcast episodes,
fetched from a dedicated authenticated endpoint
(`spclient.wg.spotify.com/transcript-read-along/v2/episode/{id}`) that is the
direct sibling of the color-lyrics endpoint we already consume. Issue #127 asks
for first-class `get_transcript(episode)` access. Like lyrics, this is a feature
that genuinely requires user credentials: the maintainer supplies `sp_dc`, the
library already mints the Web Player Bearer. Exposing transcripts as an
explicitly authenticated feature (clear `AuthenticationError` when the cookie is
missing/expired, `NotFoundError` when an episode has no transcript) reuses the
hardened lyrics machinery with no new auth code and no new dependency.

## What Changes

- New sans-io `api/transcripts.py`: URL + auth-header construction for the
  `transcript-read-along/v2/episode/{id}` endpoint, plus a pure decode of the
  response envelope into a plain mapping the parser consumes. The exact endpoint
  host/path/query lives in this one module so a future Spotify change is a
  one-line fix (the same isolation discipline as pathfinder hashes).
- New `models/transcript.py`: frozen, slotted `Transcript` / `TranscriptLine`
  on `ModelBase`, JSON-safe `to_dict()` / `from_dict()` by reflection.
- New `parse_transcript(payload)` in `api/parse_entities.py`, I/O-free, raising
  only `ParsingError`, reusing the existing private helpers.
- New `get_transcript(value)` on **both** clients (sync + async), resolving
  `"episode"`, routing through the **shared** cookie provider (the lyrics
  provider slot/builder is factored to a feature-neutral name so both features
  reuse one token), with the same 404→`NotFoundError`, 401→invalidate+retry-once
  classification lyrics uses.
- New `transcript` CLI command mirroring `lyrics`.
- `Transcript` / `TranscriptLine` exported from `spotify_scraper.models` and the
  top-level package.
- Hermetic unit tests (respx + inline body) and a `@pytest.mark.live` smoke test
  gated on `SPOTIFY_SP_DC`.

## Impact

- New: `api/transcripts.py`, `models/transcript.py`, `parse_transcript`,
  `get_transcript` on both clients, a `transcript` CLI command, exports.
- Reused unchanged: `auth/cookies.py`, `auth/totp.py`, `errors.py`,
  `models/base.py`. The lyrics provider slot is renamed to a shared name
  (internal only — no public API change).
- **One open verification item gates `api/transcripts.py`'s `decode_envelope`:**
  capture a live transcript response with the maintainer `sp_dc` to confirm the
  exact envelope shape (plain JSON vs. base64-wrapped protobuf) and where
  `language`/`isAutoGenerated` live. The model and parser are designed to be
  envelope-shape-agnostic so only `api/transcripts.py` changes once confirmed.
- Live transcript tests require a real `sp_dc` (env `SPOTIFY_SP_DC`) and are
  skipped without it.
